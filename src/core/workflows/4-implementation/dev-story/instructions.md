# Dev Story — Platform-Tracked Implementation

<critical>You MUST have already loaded and processed: the workflow.yaml for this workflow</critical>

## Overview

Executes a story implementation with the platform as the tracking system. The dev agent finds the next story from the platform (instead of local files), locks it, sets state through the workflow, and posts completion records as platform comments.

---

<workflow>

<step n="1" goal="Find the next story to implement">

**Progress: Step 1 of 7** -- Next: Load and Lock

### Role & Communication

<action>Communicate in {communication_language} with {user_name}</action>

### 1.1 Check for Review-Failed Stories

<action>First, check for issues that failed code review and need fixes:</action>
<action>Invoke the `read-context` task from `{project-root}/_aria/core/tasks/read-context.md` with `context_type: "work_items"`, `team: "{team_name}"`, `labels: ["aria-review-failed"]`, `limit: 5`</action>

<action>If aria-review-failed issues exist, prioritise those (they need fixes before new work)</action>

### 1.2 Find Stories Ready for Development

<action>If no aria-review-failed issues, find issues ready for initial development:</action>
<action>Invoke the `read-context` task with `context_type: "work_items"`, `team: "{team_name}"`, `states: ["Todo"]`, `limit: 5`</action>

<action>If user specified an issue identifier, use that instead</action>

### 1.3 Present Candidates

<action>Present candidates to the user:</action>

| # | ID | Issue | Status | Notes |
|---|---|---|---|---|
| 1 | {identifier} | Story 1.1: User Authentication | In Review | aria-review-failed: needs fixes |
| 2 | {identifier} | Story 1.2: Account Management | Todo | |

<action>
  {if autonomy_level == "interactive"}
    Wait for user selection.
  {else}
    Auto-select the first candidate (aria-review-failed issues take priority). Report selection to user.
    {if autonomy_level == "balanced"}
      "Selected {identifier}: {title}. Press Enter to confirm or type a different identifier."
    {end_if}
  {end_if}
</action>

### Success Criteria
- Candidate issues found and presented
- Issue selected (aria-review-failed prioritized)
- User confirmed selection

### Failure Modes
- Proceeding without an issue (nothing to implement)
- Not prioritizing aria-review-failed issues (old failures pile up)

</step>

<step n="2" goal="Load story details and lock">

**Progress: Step 2 of 7** -- Next: Load Context

### 2.1 Load Full Issue Details

<action>Call `get_issue` with `id: "{selected_issue_id}"` to load full issue details including description and labels</action>
<action>Call `list_comments` with `issueId: "{selected_issue_id}"` to load existing comments</action>
<action>Parse the issue description to extract: user story, acceptance criteria, tasks/sub-issues, dev notes</action>

### 2.2 Handle Review-Failed Issues

<action>If this is a aria-review-failed issue:</action>
- Parse the most recent "## Code Review Results" comment to identify findings summary
- Query for open review-finding sub-issues: invoke the `read-context` task with `context_type: "work_items"`, `parentId: "{selected_issue_id}"`, `labels: ["aria-review-finding"]`, `states: ["Backlog", "Todo", "In Progress"]`
- If open sub-issues exist, load each with `get_issue` to get severity, location, and description
- Present the dev punch list:

| # | ID | Severity | Finding |
|---|---|---|---|
| 1 | {identifier} | Critical | SQL injection in login handler |
| 2 | {identifier} | Major | Missing input validation |

- These findings are the primary focus of this implementation pass
- As each finding is addressed, set its sub-issue state to Done via `set-work-item-state` task

### 2.3 Lock and Set State

<action>Invoke `lock-work-item` task from `{project-root}/_aria/core/tasks/lock-work-item.md` with `issue_id: "{selected_issue_id}"`, `action: "lock"`, `agent_name: "dev"`</action>

<action>Invoke `set-work-item-state` task from `{project-root}/_aria/core/tasks/set-work-item-state.md` to move to In Progress:</action>

```
issue_id: "{selected_issue_id}"
target_state: "In Progress"
comment: "Dev agent starting implementation"
```

<action>If the issue is in In Review status (aria-review-failed), set state back to In Progress:</action>

```
issue_id: "{selected_issue_id}"
target_state: "In Progress"
comment: "Dev agent addressing code review findings"
```

### 2.4 Create Feature Branch (if git enabled)

<action>If `git_enabled` is `true` in module.yaml:</action>

1. **Read the platform's suggested branch name:** The `get_issue` call in step 2.1 returns a `gitBranchName` field (auto-generated by the platform from the issue identifier and title, e.g., `jacob/lim-6-story-11-user-registration`). If this field is present and non-empty, use it as the branch name.
2. **Fall back to constructed name:** If `gitBranchName` is empty or missing, invoke `git-operations` task from `{project-root}/_aria/shared/tasks/git-operations.md` with operation `create_branch`, passing `issue_key: "{issue_identifier}"` and `summary: "{issue_title}"`. This constructs a branch name from the configured prefix and kebab-cased summary.
3. **Create or check out the branch:** If using the platform's `gitBranchName`, invoke `git-operations` with operation `create_branch` and `branch_name: "{gitBranchName}"` to create or check out that exact branch. This ensures the branch name shown in the platform's UI matches the actual git branch.

<action>Log the branch name for the dev record. If the operation fails, log the error and continue -- git failures must not block development.</action>

### Success Criteria
- Issue fully loaded with description and comments parsed
- Review-failed findings identified (if applicable)
- Issue locked with aria-active label
- State set to In Progress
- Feature branch created (if git enabled)

### Failure Modes
- Not parsing aria-review-failed comments (missing fix requirements)
- Not locking before starting work (concurrent agent conflict)

</step>

<step n="4" goal="Execute implementation">

**Progress: Step 4 of 7** -- Next: Commit and Push

<action>Follow the dev-story checklist at `{checklist}`:</action>

**Critical Dev Agent Rules:**
- READ the entire story BEFORE any implementation
- Execute tasks/sub-issues IN ORDER as written
- Mark task complete ONLY when both implementation AND tests pass
- Run full test suite after each task
- Execute continuously without pausing until all tasks complete
- NEVER lie about tests being written or passing

<action>For each task/sub-issue in the story:</action>
1. Implement the code changes
2. Write comprehensive unit tests
3. Run the full test suite -- all tests must pass
4. Mark the task as complete

### Success Criteria
- All tasks executed in order
- Tests written for each task
- Full test suite passes
- No tests skipped or faked

### Failure Modes
- Implementing without tests
- Moving to next task while tests are failing
- Lying about test results

</step>

<step n="4.5" goal="Commit and push (if git enabled)">

**Progress: Step 4.5 of 7** -- Next: Post Record

This stages all changes, commits with the configured issue key prefix, and pushes if auto_push is enabled. If the operation fails, log the error and continue.</action>

### Success Criteria

- Push completed (if auto_push enabled)

</step>

<step n="6" goal="Update sub-issue statuses">

**Progress: Step 6 of 7** -- Next: Transition and Handoff

### 6.1 Query Sub-issues

<action>Invoke the `read-context` task with `context_type: "work_items"`, `parentId: "{selected_issue_id}"` to get all sub-issues</action>

### 6.2 Transition Implementation Sub-issues

<action>For implementation sub-issues (no `aria-review-finding` label): invoke `set-work-item-state` task to mark as Done</action>

### 6.3 Verify Review Finding Sub-issues

<action>For review-finding sub-issues (have `aria-review-finding` label):</action>
- Each should have been set to Done during Step 4 as findings were addressed
- Verify all review-finding sub-issues are Done
- If any remain open, warn the user: "Review finding {identifier} not yet addressed: {title}"
- Do NOT proceed to In Review state if Critical review-finding sub-issues remain open

### Success Criteria
- All implementation sub-issues set to Done
- All review-finding sub-issues verified as Done (if applicable)

</step>
